# 治理规则

## 范围

`udbx4x` 工作区维护 UDBX 的开源 SDK 与工具。

SDK 是基础。工具必须依赖 SDK，不得重复实现底层 UDBX 解析、系统表映射、GAIA 解码、CAD 解码或字段类型映射。

## 权威来源

决策优先级：

1. 当前明确的维护者决策。
2. UDBX 白皮书。
3. 真实 `.udbx` 样本证据。
4. `udbx4spec` 文档和合规资产。
5. SDK 合规测试。
6. SDK 实现代码。
7. 工具实现代码。

如果白皮书、样本和 `udbx4spec` 之间存在冲突，维护者必须先形成文档化解释，再修改 SDK 行为。

## 维护者

`udbx4x` GitHub 组织负责人拥有以下发布权限：

- `udbx4j` 的 Maven Central 发布权限。
- `udbx4ts` 的 npm 发布权限。
- `udbx4go` 的 Go module tag 发布权限。

仓库维护者可以根据仓库权限审查和合并变更；公开版本发布权限属于组织负责人或明确授权的 release maintainer。

## 项目原则

- 规范优先。
- 跨语言一致。
- SDK 先于工具。
- 不为错误概念保留兼容分支。
- 已知根因时不采用绕路补丁。
- 除非白皮书或规范规定，不把样本偶然值硬编码为通用规则。
- 公开概念和 API 变更必须文档先行。

## 决策流程

### 格式或 API 变更

格式、API、DatasetKind、FieldType、几何模型或错误分类变更必须从 `udbx4spec` 开始。

必要顺序：

1. 更新或新增规范文本。
2. 在需要时新增或更新合规夹具。
3. 更新受影响的 SDK。
4. 更新测试和合规报告。
5. 更新用户文档。

### SDK 变更

SDK 变更必须保持共享模型一致，除非已经接受明确的规范变更。

允许语言生态惯用表达，但语义行为必须一致。

### 工具变更

工具变更必须使用 SDK API。若工具需要缺失的 SDK 能力，应先给 SDK 增加能力，而不是在工具中复制解析逻辑。

## 兼容承诺

Patch release 修复缺陷，不改变预期公开 API 行为。

Minor release 可以增加 API 和格式能力。

Major release 可以在当前 API 与规范冲突或阻碍长期正确性时进行破坏性变更。

项目不为过时概念保留兼容分支。破坏性变更必须提供迁移说明。

## 文档要求

任何用户可见行为变更至少更新以下文档之一：

- README。
- CHANGELOG。
- API 文档。
- 规范。
- 合规报告。

任何公开概念或术语变更，在 `docs/concepts/glossary.md` 建立后必须同步更新术语表。
