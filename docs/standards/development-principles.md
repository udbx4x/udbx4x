# 开发原则与编码规范

本文档定义 `udbx4x` 工作区的通用开发原则。语言特定规范可以在子项目中补充，但不得与本文档冲突。

## 基本原则

- 概念统一。
- 遵从规范。
- 文档优先。
- 架构优雅。
- 代码简洁。
- 回归本质。
- 避免重复。
- 拒绝硬编码。
- 效率优先。
- 充分沟通。

## 规范优先

公开 API、格式解释、数据模型、枚举值、字段映射和错误分类变更，必须先更新 `udbx4spec`。

不得直接在某个 SDK 中引入与规范不一致的新概念。

## 不保留错误兼容

本项目不为了旧错误实现保留兼容分支。

禁止：

- 为错误概念增加长期兼容层。
- 用打补丁方式绕过根因。
- 在工具层复制 SDK 内部解析逻辑。
- 把样本文件中的偶然值硬编码为格式规则。

如果旧实现与规范冲突，应修正旧实现，并通过版本说明表达破坏性变更。

## 代码简洁

实现应直接、清楚、可测试。

优先：

- 小函数。
- 单一职责。
- 明确命名。
- 显式错误处理。
- 清楚的数据边界。

避免：

- 过早抽象。
- 无实际收益的接口层。
- 为假想场景设计复杂扩展点。
- 混入无关重构。

## 根因修复

修复 bug 时必须尽量定位根因。

标准流程：

1. 复现问题。
2. 判断是规范缺失、实现错误、样本异常、测试不足还是文档误导。
3. 补充能覆盖根因的测试或合规夹具。
4. 修复实现。
5. 更新文档或风险记录。

不接受只让当前样本通过的临时修复。

## 测试要求

测试优先级：

1. `udbx4spec` 合规夹具和 golden bytes。
2. 跨语言读写一致性测试。
3. 子项目单元测试。
4. 子项目集成测试。
5. 工具端到端测试。

新增格式能力必须有测试。新增公开 API 必须有测试。修复缺陷应有回归测试。

## 注释规范

代码注释只解释不明显的原因、格式来源或边界条件。

适合注释：

- 白皮书字段来源。
- 二进制布局。
- 安全边界检查。
- 跨语言兼容原因。

不适合注释：

- 重复代码表面含义。
- 描述显而易见的赋值。
- 为临时绕路找理由。

## 提交规范

建议使用简洁 commit message：

```text
feat: add pointZ golden byte fixture
fix: update SmObjectCount after delete
test: add malformed GAIA blob regression
docs: clarify DatasetKind mapping
refactor: simplify field type conversion
chore: update release checklist
```

## 完成自检

提交前检查：

- 是否需要先修改 `udbx4spec`。
- 是否符合根目录 `AGENTS.md` 的语言和文档约定。
- 是否补齐测试。
- 是否更新 README、CHANGELOG 或合规报告。
- 是否引入重复实现。
- 是否引入硬编码。
- 是否解决根因。
